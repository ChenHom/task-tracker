# 主工作區討論收斂流程設計

**日期：** 2026-07-13  
**範圍：** 固定主協作工作區 `11a82028-fc50-466a-a723-e002032cd9a6` 的非規則討論 TASK

## 宗旨

主工作區是目前七位成員共同提出與討論工作問題、改善方向及優化想法的地方。現有成員為唯一 OWNER `user01@test.local`，以及 `user02-06@test.local`、`user09@test.local` 六位 Commenter。七位成員都能建立討論 TASK 與留言；後續是否加入其他成員不在本次範圍。

主工作區只負責討論、形成共識與交接，不承載實作。討論 TASK 只走 `Todo -> Done`，永遠不進入 `Doing` 或 `Review`。需要實作時，OWNER 必須在對應工作區另建 TASK。

## 建立 TASK

主工作區的 TASK 可由目前任何一位成員建立，不限制特定人員。TASK 適用於工作中產生的問題、阻礙、風險、改善建議或優化想法。

建立畫面的描述輸入框預先填入以下範本，協助建立者整理內容；這是可編輯範本，不做後端必填驗證：

```text
問題／優化想法：
目前情況或影響：
希望改善的結果：
```

建立後仍維持既有描述編輯能力。描述更新不鎖定、不自動留言、不重設討論期限；OWNER 若漏看更新內容，由建立者自行負責提醒。

## OWNER 啟動討論

OWNER 應在 TASK 建立後 24 小時內完成以下兩則留言。超過 24 小時不禁止 OWNER 後續啟動討論；此時限由 OWNER 與 sweep 流程遵循，前端不顯示尚未啟動或逾時提醒。

### 1. OWNER 先提出完整看法

OWNER 必須先留下獨立的 `【OWNER想法】` 留言，不可與全員通知合併。內容至少使用以下結構：

```text
【OWNER想法】
現況／問題：
預期價值：
風險與反對理由：
現行可替代方案：
初步判斷：
希望成員確認的問題：
```

OWNER 的責任是全面思考，不預設建立者的想法一定正確或值得實作。留言必須同時評估正面價值、負面影響、可行性、風險，以及現行是否已有足以替代的方案。

### 2. OWNER 發出全員回覆通知

OWNER 接著使用既有留言輸入框，手動 `@user02-06` 與 `@user09` 全部六位 Commenter，並以 `【全員回覆：N天】` 開頭要求全員補充或表態。OWNER 不需要 `@` 自己；固定名單不另做漏人驗證。

通知範例：

```text
【全員回覆：2天】
@成員一 @成員二 @成員三 @成員四 @成員五 @成員六
請針對上述方向提出補充、疑慮或替代方案；沒有補充也請留言表示已閱讀。
```

若期限超過 `2` 天，通知需再加入非空白的 `較長期限理由：`。

這則通知送出時才開始計時。

## 回覆與討論期限

- 最短 `2` 天，最長 `7` 天。
- 以半天為最小單位，可使用 `2`、`2.5`、`3` 至 `7` 天。
- 一天等於連續 `24` 小時，半天等於連續 `12` 小時；不是工作日，不因週末或假日順延。
- 預設使用最短的 `2` 天，讓新問題盡快完成討論。
- 只有近期成員已有大量事務需要處理時，OWNER 才使用超過 `2` 天的期限，並在同一則通知說明原因。
- 通知送出後期限固定，不可延長、縮短或重新開窗。
- 即使全員提前回覆並形成共識，原 TASK 仍保持 `Todo` 到截止時間。

所有 Commenter 都應在通知後留言；沒有補充時也應明確表示已閱讀。系統不建立缺席名單、不判定逾時回覆，也不因任何人未回覆而阻擋收尾。OWNER 直接從留言作者與內容判斷參與情況，未回覆者不再另行處理。

## 共識與確認

OWNER 可在等待期間依討論內容提出結論，不必等到截止後才開始整理。

### 一般建立者

OWNER 先留下 `【結論】` 或 `【結論：不實作】`；TASK 建立者必須在該留言之後，以 `【確認結論】` 明確同意。只有這兩方依序確認，才算形成共識，不能把 OWNER 的單方看法當作共同結論。

### OWNER 自己建立的 TASK

若 OWNER 同時是 TASK 建立者，則由任一位 Commenter 在 OWNER 結論後留下 `【確認結論】`。沒有 Commenter 確認，就不算形成共識。

## 截止後的三種結果

截止時間到達後不自動改變狀態，由 OWNER 在下一次操作或巡檢時完成收尾。未回覆成員不阻擋以下任何結果。

### 1. 有共識且需要實作

1. OWNER 與建立者／共同確認者已完成雙方結論。
2. OWNER 在對應的 canonical／目標工作區建立新的實作 TASK；不得把實作 TASK 留在主工作區。
3. OWNER 在原討論留下純文字交接，不提供 URL 或可點擊連結：

```text
【實作任務】工作區：<工作區名稱>｜TASK：<TASK 名稱>
```

4. OWNER 將原討論由 `Todo` 直接設為 `Done`。

### 2. 有共識但不實作

若雙方確認現行已有可用替代方案、目前不值得投入或不需要修改，OWNER 留下 `【結論：不實作】`，由建立者／共同確認者以 `【確認結論】` 確認後，在截止時間將原討論直接設為 `Done`，不建立新的實作 TASK。

### 3. 未達共識

如果截止時沒有雙方確認，OWNER 留下以下內容後將原討論直接設為 `Done`，不建立實作 TASK：

```text
【未達共識】
尚未解決的分歧：
缺少的確認或資訊：
下次重新思考前的建議：
```

建立者不必再確認「未達共識」。日後有人重新思考並取得新資訊時，另開一筆新的主工作區討論 TASK，不重新開啟或延長舊 TASK。

## 系統行為與最小資料設計

### 討論窗口

新增 `main_discussion_windows`，每個主工作區討論 TASK 最多一筆：

- `task_id`
- `owner_thought_comment_id`
- `request_comment_id`
- `opened_at`
- `wait_half_days`，允許 `4` 到 `14`，分別代表 `2` 到 `7` 天
- `due_at`

當 OWNER 發出合法的 `【全員回覆：N天】` 留言時，後端確認先前已有 OWNER 的 `【OWNER想法】`，解析整數或半天單位的 `N`，並以同一交易保存留言與固定窗口。`due_at = opened_at + wait_half_days * 12 小時`。超過建立後 24 小時仍可開窗，不新增前端逾時狀態。

不新增回覆追蹤表、缺席名單、逾時回覆狀態或自動通知。既有留言本身就是討論紀錄。

### 收尾守門

主工作區非規則討論只允許 OWNER 執行 `Todo -> Done`。後端拒絕 `Doing`、`Review`、從 `Done` 回退或任何其他狀態轉換。

OWNER 在截止前嘗試設為 `Done` 時，後端拒絕並回傳截止時間。截止後則依留言順序辨識以下任一合法收尾：

- `【結論】` -> 建立者／共同確認者 `【確認結論】` -> OWNER `【實作任務】工作區：...｜TASK：...`
- `【結論：不實作】` -> 建立者／共同確認者 `【確認結論】`
- OWNER `【未達共識】`，且分歧、缺少資訊與下次建議皆有內容

成功時追加單一 `task.main_discussion_concluded` event，payload 包含：

- `status: 'Done'`
- `outcome: 'implement' | 'no_implementation' | 'no_consensus'`
- `windowOpenedAt`、`windowDueAt`
- 對應的結論／確認／交接 comment id
- `implement` 時的工作區名稱與 TASK 名稱

projection 只把原討論更新為 `Done`，不指派 OWNER。一般工作區仍沿用既有 `Todo -> Doing -> Review -> Done` 狀態機。

### 前端

- 主工作區建立 TASK 時，描述框預填建立範本。
- 不新增等待天數選擇器；OWNER 直接在留言中寫 `N天`。
- 主工作區狀態控制只提供 `Todo` 與 `Done`；後端仍是最終權限與流程權威。
- 前端不顯示討論截止時間、是否逾期、回覆清單或缺席名單，也不在留言框旁新增格式提示；OWNER 直接依政策文字與留言內容操作。

## OWNER sweep 與政策文件

OWNER sweep prompt 必須反映完整流程：

1. 新討論先全面評估正反面與現行替代方案。
2. 24 小時內先發 `【OWNER想法】`，再 `@` 六位 Commenter 發出最短可行期限的全員通知。
3. 等待期間推動 OWNER 與建立者形成雙方結論。
4. 截止前不移動 TASK；截止後依實作、不實作或未達共識三條路徑直接完成為 `Done`。
5. 不記錄或追逐缺席者，不在主工作區建立實作 TASK。

同步更新主工作區政策文字、`design.md`、`docs/operations.md` 與 `docs/tasks/current.md`。這些變更不授權執行 `npm run sim`、任何 live sweep 或啟用 timer。

## 錯誤與例外

- 非 OWNER 不能開啟窗口、提出系統認可的 OWNER 結論或改變主工作區狀態。
- 規則 TASK、非主工作區 TASK、非 `Todo` TASK 或已開窗 TASK 不可建立窗口。
- `N` 必須介於 `2` 到 `7` 天，且只能以 `0.5` 天遞增；其他格式視為一般留言，不開始計時。
- 超過 `2` 天的通知需有非空白的 `較長期限理由：`；內容品質由 OWNER 負責。
- 工作流程使用 UTC ISO timestamp 保存與比較；前端不新增期限或逾時呈現。
- 留言修改與刪除沿用既有權限；收尾前缺少必要標記時不能 `Done`，收尾後 event 已保存的結果不回寫。
- 描述更新不影響窗口、期限或收尾判定。

## 測試與驗收

### Domain／資料

1. 七位現有成員都能建立主工作區 Todo 討論；一般建立規則與 Commenter 權限維持不變。
2. OWNER 需先有結構完整的 `【OWNER想法】`，才可用 `2` 到 `7` 天、半天遞增的通知開窗。
3. `due_at` 以連續 12 小時單位精確計算；週末與假日不影響。
4. 窗口不可延長、縮短或重開；開窗超過建立後 24 小時仍不阻擋，也不新增前端逾時狀態。
5. 主工作區只允許 OWNER 的 `Todo -> Done`；截止前及其他狀態轉換全部拒絕。
6. 一般建立者與 OWNER 自建 TASK 的雙方確認規則分別驗證。
7. 實作、不實作、未達共識三種收尾皆產生正確的 `task.main_discussion_concluded` event。
8. 沒有成員回覆仍可在截止後走未達共識並完成；不產生缺席資料。
9. 一般工作區的既有四狀態流程不受影響。

### HTTP／前端

1. 主工作區建立畫面預填描述範本，且仍可自由修改。
2. 依序驗證 OWNER 想法、全員通知、截止前 `Done` 被拒、截止後三種合法收尾。
3. 驗證 Commenter 無法改狀態，OWNER 也無法把主工作區討論改為 `Doing` 或 `Review`。
4. 驗證畫面不新增等待選擇器、截止／逾時資訊、回覆清單、缺席名單或留言格式提示。
5. 驗證實作交接只顯示工作區名稱與 TASK 名稱，不產生 URL。

預設驗證依專案規範執行 `npx tsc --noEmit`、直接相關的 `npx tsx src/<file>.test.ts`、完整 `npm test`、`npm run build` 與 `git diff --check`。不執行 live AI run。
