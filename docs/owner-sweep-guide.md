# Owner 啟動與巡檢守則

本文件定義 `user01`（Owner）在人工啟動、人工巡檢或經授權的自動 sweep 時應遵守的工作方式。它是可調整的團隊作業規範；API 權限、狀態機與部署操作仍以 [API 文件](api.md) 與 [操作手冊](operations.md) 為準。

## 角色目的

Owner 的責任是讓工作可被推進、驗收、結案，或明確停在可追蹤的阻塞狀態；不是代替成員完成所有實作。巡檢以具體決策與下一步為優先，不為清空看板而改變狀態。

## 兩層巡檢

### 每次都做：核心看板治理

1. **盤點工作區與 task**：讀取主協作工作區的待處理討論，以及一般工作區的 `Todo`、`Doing`、`Review`。忽略 `[規則] 主工作區協作與交接`，它是政策提示，不是工作項目。
2. **處理主協作討論**：新討論在建立後盡量 24 小時內先留下完整的 `【OWNER想法】`，再用 `【全員回覆：N天】` 開啟固定回覆窗口。預設 `N=2`；只有成員近期工作量確實需要時才用較長期限並寫明理由。期限前不變更 task 狀態。
3. **讓一般 task 有下一步**：每個未完成 task 都應能回答「誰做下一步」、「下一個可驗收成果是什麼」及「何時需要 Owner 再判斷」。`Doing` 沒有進度時催辦或釐清阻塞；無主 `Todo` 補足範圍與驗收條件，讓成員可認領。
4. **驗收與調整狀態**：一般 task 遵守 `Todo → Doing → Review → Done` 的相鄰狀態機；一次 PATCH 只改一個欄位。驗收證據完整且 CI 為 PASS 才可完成。CI 為 SKIP 時，Owner 必須審閱 diff、task 驗收證據及實際檢查後再決定；CI 為 FAIL 或證據不足時，留言指出具體缺口並退回 `Doing`。
5. **管理阻塞與異常**：遇到權限、環境、工具、跨 repo 或驗收阻塞，留下「原因、負責人或外部依賴、下一步」。沒有新資訊時不重複完整診斷；維持原狀並在有變化時更新即可。
6. **提出想法**：在協助成員、驗收或自身工作中發現值得討論的改善時，到主協作工作區建立 `[討論]` task。描述至少包含問題、影響、預期結果與初步方向；不得直接把未經討論的想法當成實作 task。

### 僅在需要時做：作業前置檢查

只有在經授權執行 live sweep、服務異常、準備驗收或合併、或將進行會修改看板的自動作業時，才檢查：

- `task-tracker.service` 與 `/api/health`；
- `sim-logs/.run.lock` 是否被活躍流程持有；
- Owner runner、成員 runner 與額度資料是否可用；
- 相關 repo／worktree 是否位於正確情境且可驗證。

不得因巡檢自行啟用 timer 或執行 live sweep；這些動作仍需要明確人工授權。

## 主協作工作區的收斂

主協作工作區只放討論，不直接實作。固定窗口到期後，只有 Owner 可以依完整留言證據將 `Todo` 直接收斂為：

- **要實作**：寫下 `【結論】`、取得建立者或共同確認者的 `【確認結論】`，在目標工作區建立實作 task，並記錄 `【實作任務】工作區：...｜TASK：...`；
- **不實作**：留下 `【結論：不實作】` 與 `【確認結論】`；
- **未達共識**：留下 `【未達共識】`，並完整寫出分歧、缺少資訊與下次建議。

主協作工作區不使用 `Doing` 或 `Review`，不可在期限前結案，也不可從 `Done` 回退。

## 工作區結案與封存

一般工作區只有在所有一般 task 都是 `Done` 或 `Archived`、且沒有尚待收斂的主協作討論時，才可進行封存。封存前：

1. 確認沒有仍需驗收、阻塞或待處理的工作；
2. 依既有權限規則移除所有非 Owner 的 active 成員，使工作區只剩 Owner；
3. 封存工作區，並在有必要時留下結案／封存原因。

固定的主協作工作區永不適用這個封存流程。封存不是刪除；已封存工作區仍保留歷史與審計資料。

## 留言與紀錄原則

只在有實質變化時留言：派工或補充範圍、狀態變更、驗收結果、阻塞、封存，或新想法。沒有變化的巡檢不新增摘要，避免把 task 留言淹沒。

固定期限尚未到期、既有共識未變且沒有新實質意見時，Owner 不新增「仍為 Todo」或截止提醒；下一次有實質變化或必須收斂時再更新。

每次留言應讓下一位讀者知道目前決策、可驗證證據與下一步。無法立即處理的事項也必須留下可追蹤的下一步，而不是靜默停留在原狀。
